# Resume Engine v1.4

## Purpose

Resume Engine v1 powers ScoreLab's English-only resume analysis experience.

It supports:

- Resume analysis
- Scoring
- Free preview
- Paid improvement plan
- Downloadable report data
- Future resume-related upgrades

The engine is modular and should stay separated from React page components.

Primary entry point:

```ts
analyzeResumeWithEngine({
  resumeText,
  targetRole
});
```

- Engine name: Resume Engine

## Scoring Categories

Total score: 100 points.

Each category is scored from 0-20.

1. ATS Compatibility - 20 points
2. Clarity & Structure - 20 points
3. Impact & Achievements - 20 points
4. Keyword Relevance - 20 points
5. Professional Presentation - 20 points

Each category includes:

- score
- grade
- explanation
- evidence from resume
- improvement advice

## Rulebook

### ATS Compatibility

- Avoid tables, columns, images, heavy graphics, and visual-only resume elements.
- Use standard section headings.
- Include clear job titles.
- Include readable dates.
- Include standard contact information.
- Use ATS-friendly keywords naturally.

### Clarity & Structure

- Resume should have a clear professional summary when useful.
- Work experience should be easy to scan.
- Most relevant experience should appear first.
- Bullet points should be concise.
- Sections should follow a logical order.

### Impact & Achievements

- Bullet points should show outcomes, not only responsibilities.
- Strong bullets should include numbers, metrics, scope, or business impact when truthful.
- Avoid vague phrases like "responsible for".
- Use action verbs.
- Prioritize achievements over tasks.

### Keyword Relevance

- Skills should match the target role when provided.
- Technical and functional keywords should be visible.
- Repeated important keywords are acceptable if natural.
- Missing role-specific keywords should be identified.
- Generic skills should not dominate the resume.

### Professional Presentation

- Tone should be professional.
- Grammar and spelling should be checked.
- Formatting should be consistent.
- Resume should avoid unnecessary personal information.
- Resume should feel focused, credible, and easy to read.

## Resume Engine v1.1 Rule System

Resume Engine v1.1 upgrades the rulebook from plain category guidance into structured rule objects.

Each rule includes:

- `id`
- `category`
- `title`
- `description`
- `priority`
- `weight`
- `evidenceType`
- `recommendation`
- `exampleFix`

Rule ID prefixes:

- `ATS`: ATS Compatibility
- `CLR`: Clarity & Structure
- `IMP`: Impact & Achievements
- `KEY`: Keyword Relevance
- `PRS`: Professional Presentation

Priorities:

- `Critical`
- `Major`
- `Minor`

The v1.1 rulebook includes 50 structured rules:

- 10 ATS Compatibility rules
- 10 Clarity & Structure rules
- 10 Impact & Achievements rules
- 10 Keyword Relevance rules
- 10 Professional Presentation rules

The AI must include triggered rules in paid output:

```ts
triggeredRules: {
  ruleId: string;
  ruleTitle: string;
  priority: "Critical" | "Major" | "Minor";
  evidence: string;
  explanation: string;
  recommendation: string;
}[]
```

Free preview should not expose triggered rules directly. It should only show the top 3 issues.

## Resume Engine v1.2 Deterministic Pre-Checks

Resume Engine v1.2 adds deterministic pre-checks before AI analysis.

The purpose is to reduce AI randomness and make resume scoring more evidence-based, consistent, and trustworthy.

Pre-checks run in:

```text
lib/engines/resume/resumePrechecks.ts
```

Main function:

```ts
runResumePrechecks(resumeText, targetRole)
```

The pre-check module detects:

- word count
- character count
- email presence
- phone presence
- LinkedIn presence
- portfolio or website presence
- summary section
- experience section
- education section
- skills section
- certifications section
- detected section headings
- estimated bullet count
- quantified bullet count
- weak phrase count
- action verb count
- date pattern count
- possible formatting warnings
- extraction quality warnings
- precheck-triggered rules

Basic deterministic issues include:

- missing email
- missing phone
- missing LinkedIn
- resume text too short
- no clear work experience section
- no clear skills section
- no clear education section
- very few bullet points
- few or no quantified achievements
- repeated weak phrases such as "responsible for", "helped with", "worked on", "involved in", and "participated in"
- possible extraction issues from broken lines or unreadable characters

Pre-check findings map to existing rule IDs where possible, such as:

- `ATS-003` for contact information
- `ATS-008` for extraction/readability issues
- `ATS-010` for missing skills section
- `CLR-004` for missing work experience section
- `CLR-005` for weak bullet structure
- `CLR-009` for missing education section
- `IMP-002` for missing quantified achievements
- `IMP-003` for weak responsibility phrases

The AI receives:

- resume text
- target role
- structured rulebook
- deterministic prechecks

The AI must use pre-checks as evidence and should not contradict deterministic findings unless the resume text clearly justifies it.

## Resume Engine v1.3 Quality Calibration

Resume Engine v1.3 combines deterministic evidence anchors with AI recruiter judgment.
Deterministic anchors stabilize category scores; AI judgment refines the score and writes
personalized explanations.

Score bands:

- 0-30: Poor
- 31-55: Weak
- 56-75: Average
- 76-90: Strong
- 91-100: Excellent

Interview readiness uses:

- Low
- Medium
- High

Resume intelligence now detects metrics anywhere in extracted text, including percentages,
revenue, savings, users, customers, project volume, team size, delivery time, and performance
improvements. It also detects leadership, ownership, communication, technical depth, business
impact, and role-focus signals.

Contact completeness contributes to ATS readiness but does not dominate the score or occupy
multiple preview issues. Preview issues prioritize role clarity, evidence, relevance, and
readability, and are semantically deduplicated.

DOCX extraction preserves paragraph and list structure. PDF extraction preserves line breaks
and supports ASCII85 plus Flate encoded text streams when the primary parser cannot read a
valid text PDF.

When no target job is supplied, Keyword Relevance measures internal role-family coherence and
evidenced skills. It must not be represented as a true job-match score.

The report includes:

```ts
precheckSummary
precheckTriggeredRules
paidReport.resumeEnginePreCheck
```

The paid report pre-check section stays concise:

- contact completeness
- section completeness
- quantified achievements
- weak phrases
- extraction quality warnings

- Top 10 improvement priorities

## Error Handling

The engine handles:

- Empty resume text
- Very short resume text
- Invalid JSON from AI
- Missing required schema fields
- OpenAI API failure

User-facing errors should be clear and non-technical.

## OpenAI Production Mode

Resume Engine analysis uses the real OpenAI API and returns structured JSON that matches the report schema.

If OpenAI is unavailable, missing, or returns invalid output, the engine should return a clear error instead of generating a fake report.

## Resume Upload File Support

Supported upload formats:

- PDF
- DOCX
- DOC
- JPG
- JPEG
- PNG

Priority:

1. Stable PDF
2. Stable DOCX
3. Graceful JPG/PNG support
4. DOC support only when practical

Current extraction behavior:

- PDF uses the existing `pdf-parse` flow with best-effort fallback.
- DOCX uses lightweight DOCX XML text extraction.
- DOC returns a clear fallback message when extraction is not practical.
- JPG/PNG image upload is experimental. If OCR is not enabled, users are asked to upload PDF or DOCX.

Upload limits:

- Maximum file size: 10MB

User-facing errors should clearly explain:

- unsupported file type
- file too large
- text extraction failure
- OCR fallback
- too little readable text
